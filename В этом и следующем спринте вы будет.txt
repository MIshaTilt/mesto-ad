Список предустановленных карточек берётся из массива initialCards, расположенного в файле src/scripts/cards.js.
Все функции хранятся в директории src/scripts/components/ и разбиты по отдельным файлам в зависимости от их функциональности.
Функции для работы с карточками вынесены в модуль src/scripts/components/card.js:
likeCard — переключает состояние лайка на кнопке: устанавливает лайк, если его нет, и удаляет, если он уже есть;
deleteCard — удаляет карточку;
getTemplate — клонирует шаблон #card-template и возвращает разметку новой карточки;
createCard — возвращает готовую разметку карточки на основе переданных данных и добавляет слушатели событий:
клик по кнопке с иконкой сердечка переключает состояние лайка,
клик по кнопке с иконкой корзины удаляет карточку,
клик по изображению открывает его в полноразмерном виде в отдельном модальном окне.
Функции для работы модальных окон вынесены в модуль src/scripts/components/modal.js:
handleEscUp — обработчик события нажатия клавиши Escape, закрывает открытое модальное окно при нажатии клавиши;
openModalWindow — открывает модальное окно и добавляет обработчик нажатия Escape;
closeModalWindow — закрывает модальное окно и удаляет обработчик нажатия Escape;
setCloseModalWindowEventListeners — устанавливает слушатели событий для закрытия модального окна.
Файл src/scripts/index.js — точка входа в приложение. В нём находятся:
вызовы функций, импортированных из модулей;
объявления и инициализация глобальных констант и переменных с DOM-элементами страницы;
обработчики событий: открытия модальных окон редактирования профиля, изменения аватара, добавления карточки и просмотра полноразмерного изображения карточки; также обработчики события отправки форм — редактирования данных, аватара пользователя и создания карточки;
вывод предустановленных карточек на страницу;
установка слушателей закрытия всех модальных окон.

Настройка сборки с помощью Vite
Первое, что вам понадобится сделать, — подключить сборщик Vite к проекту.
Для начала откройте терминал и выполните команду node -v . Убедитесь, что в системе установлена версия node.js выше 20.19. Если версия ниже, обновите её до актуальной, следуя инструкции на официальном сайте Node.js.
Затем инициализируйте проект, установите сборщик Vite и добавьте в файл package.json в раздел scripts команды: npm run dev — для запуска, npm run build — для сборки проекта. Теперь вы сможете запускать dev-сервер для разработки и делать сборку проекта.
После этого создайте в корне проекта файл vite.config.js и настройте сборщик таким образом, чтобы при запуске dev-сервера страница проекта автоматически открывалась в браузере.
Валидация форм
В блоке профиля пользователя есть две кнопки.
Кнопка с иконкой карандаша открывает модальное окно с формой изменения данных пользователя. Обратите внимание, что поля формы при открытии заполняются текущими данными, взятыми со страницы.

Кнопка с иконкой плюса открывает модальное окно с формой добавления новой карточки. Чтобы добавить новую карточку, нужно заполнить поля с названием карточки и ссылкой на изображение.

А ещё при клике на аватар открывается модальное окно с формой изменения аватара.

У этих форм отсутствует кастомная валидация. Именно это нам и предстоит исправить.
Подготовка разметки форм
На первом этапе ваша задача — подготовить разметку форм, добавив необходимые HTML-элементы и атрибуты для валидации. 
Сначала отключите встроенную браузерную валидацию форм и добавьте для кнопок форм класс popup__button_disabled и атрибут disabled.
Валидация формы «Редактировать профиль»
Если поле формы «Редактировать профиль» не прошло валидацию, под ним должен появиться красный текст ошибки.
Настройки валидации следующие:
Оба поля обязательные.
Поле popup__input_type_name должно содержать от 2 до 40 символов и может включать только латинские и кириллические буквы, знаки дефиса и пробелы. Для проверки символов нужно использовать регулярное выражение.
Поле popup__input_type_description должно содержать от 2 до 200 символов.
Если какое-либо из полей не проходит валидацию, необходимо вывести кастомное сообщение об ошибке. В случае, если ошибка не попадает под правила валидации, следует отображать стандартный текст ошибки, предоставляемый браузером.
Для хранения кастомного текста ошибки поля popup__input_type_name добавьте ему атрибут data-error-message="Разрешены только латинские, кириллические буквы, знаки дефиса и пробелы". 
Для отображения текста ошибок добавьте после полей ввода элементы:
<span class="popup__error" id="user-name-error"></span> — для поля popup__input_type_name;
<span class="popup__error" id="user-description-error"></span> — для поля popup__input_type_description.
Валидация формы «Новое место»
Задание аналогично предыдущему, но есть отличия:
Не нужна проверка длины текста у поля ссылки.
Нужна проверка того, что пользователь ввёл именно ссылку.
Настройки валидации следующие:
Оба поля обязательные.
Поле popup__input_type_card-name должно содержать от 2 до 30 символов и может включать только латинские и кириллические буквы, знаки дефиса и пробелы. Как и ранее, для проверки символов нужно использовать регулярное выражение, а кастомный текст ошибки хранить в атрибуте data-error-message. Ошибку выводите в элемент <span class="popup__error" id="place-name-error"></span>. Если ошибка не попадает под правила валидации, отображайте стандартный текст ошибки, предоставляемый браузером.
Поле popup__input_type_url должно содержать ссылку. Ошибку выводите в элемент <span class="popup__error" id="place-link-error"></span>.
Валидация формы «Изменения аватара»
Поле обязательное и должно содержать ссылку. Ошибку выводите в элемент <span class="popup__error" id="user-avatar-error"></span>.
Требования к модульности
Прежде чем приступить к написанию JavaScript-кода валидации, важно разобраться с требованиями к модульности проекта.
В приложении много разных функций. Если собрать их в одном файле src/scripts/index.js, читать его будет сложно. Поэтому код разделён на разные модули. 
Несколько правил:
В модуле src/scripts/components/card.js не должно быть функций, отвечающих за обработку формы создания карточки. В нём должны быть только те функции, в которых производится работа с разметкой самой карточки, её клонирование, заполнение, удаление, установка или снятие лайка.
Модуль src/scripts/components/modal.js отвечает только за логику открытия/закрытия модального окна. Он тоже не должен содержать функции, отвечающие за обработку формы.
В файле src/scripts/index.js должны находиться объявления и инициализация глобальных констант и переменных с DOM-элементами страницы, обработчики событий открытия и закрытия модальных окон и отправки форм, вызовы других функций, подключённых из модулей, которым нужно будет передавать объявленные здесь переменные и обработчики.
Писать код, отвечающий за валидацию, нужно в отдельном модуле src/scripts/components/validation.js. Код модуля должен быть написан так, чтобы его можно было забрать из проекта и перенести в другой без необходимости что-то менять внутри. Подумайте, какие настройки для этого нужно передать функциям в качестве аргументов. Из этого модуля должны экспортироваться только функции включения валидации и очистки ошибок.
Требования к коду валидации
Разбейте код валидации на отдельные функции:
showInputError — отображает сообщение об ошибке под невалидным полем и добавляет соответствующие классы.
hideInputError — скрывает сообщение об ошибке и удаляет классы, связанные с ошибкой.
checkInputValidity — проверяет валидность конкретного поля. Если оно невалидно, вызывает showInputError, иначе — hideInputError. В случае, если в поля «Имя» или «Название» введён любой символ, кроме латинской буквы, кириллической буквы и дефиса, выводит кастомное сообщение об ошибке: «Разрешены только латинские, кириллические буквы, знаки дефиса и пробелы». Текст ошибки разместить в data-* атрибуте поля ввода.
hasInvalidInput — возвращает значение true, если хотя бы одно поле формы не прошло валидацию.
disableSubmitButton — делает кнопку формы неактивной.
enableSubmitButton — включает кнопку формы.
toggleButtonState — включает или отключает кнопку формы в зависимости от валидности всех полей. Если хотя бы одно из полей не прошло валидацию, кнопка формы должна быть неактивной. Если оба поля прошли — активной.
setEventListeners — добавляет обработчики события input для всех полей формы. При каждом вводе проверяет валидность поля и вызывает функцию toggleButtonState.
clearValidation — очищает ошибки валидации формы и делает кнопку неактивной.
Принимает DOM-элемент формы и объект с настройками. Используйте эту функцию при открытии формы редактирования профиля.
enableValidation — отвечает за включение валидации всех форм. Функция должна принимать все нужные функциям селекторы элементов как объект настроек.

// файл index.js
import { enableValidation, clearValidation } from "./components/validation.js";

// Создание объекта с настройками валидации
const validationSettings = {
  formSelector: ".popup__form",
  inputSelector: ".popup__input",
  submitButtonSelector: ".popup__button",
  inactiveButtonClass: "popup__button_disabled",
  inputErrorClass: "popup__input_type_error",
  errorClass: "popup__error_visible",
};

// включение валидации вызовом enableValidation
// все настройки передаются при вызове
enableValidation(validationSettings); 
Напоминаем: в модуле валидации должен быть только универсальной код. Никаких конкретных селекторов для кнопок и полей ввода быть не должно. Все селекторы для валидации нужно брать только из объекта валидации, который передаётся в вызов каждой функции, начиная с enableValidation(settings), затем в setEventListeners(formElement, settings) и так далее.
Что важно помнить
При открытии модального окна редактирования профиля в поля формы подставляются данные пользователя со страницы.
После добавления карточки на страницу или изменения аватара форма очищается, при повторном открытии модального окна поля форм должны быть пустыми.
Если открыть модальное окно, ввести в поля ввода формы невалидные данные и закрыть окно, то при повторном открытии необходимо вызвать очистку ошибок валидации, которые могли остаться с прошлого открытия.
При первом открытии формы, пока не внесены изменения, кнопка формы должна быть не активна.
Заключение
Если перезагрузить страницу, то все изменения пропадут, так как сейчас мы нигде не сохраняем информацию. Но дальше, в финальном проекте, мы постепенно реализуем всю функциональность страницы и интегрируем её с бэкендом.
В реальных проектах так и происходит: фронтенд и бэкенд пишутся отдельно, а потом их связывают друг с другом.
